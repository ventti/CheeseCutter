# CheeseCutter-Extended ‚Äî notes for Claude

## Orienting yourself (read this first)

**At the start of any task that involves the codebase, before grepping or
reading source widely:**

1. **Refresh the repo index:** run `make map` (or `python3 tools/genmap.py` if
   you can't invoke make) ‚Äî it's a fast, build-free regen that keeps the index
   honest against the current source.
2. **Read `doc/ARCHITECTURE.md` for context** to locate the right module(s) and
   see what depends on what, *then* open source. Don't reconstruct the layout by
   hand when the index already has it.

Two low-maintenance aids exist so you don't have to grep the whole tree to learn
the layout:

- **`doc/ARCHITECTURE.md`** ‚Äî a compact, **generated** map of every first-party
  module: package roles, a one-line synopsis per module, and the internal
  dependency edges. Read it first to find where things live and what depends on
  what. Its YAML frontmatter restates how to use and update it. It is produced by
  `tools/genmap.py` ‚Äî **do not hand-edit**; run `make map` (also part of
  `make docs` and `mise run build`) to regenerate.
  - The only hand-written input is each module's **one-line synopsis**, taken
    from the first sentence of the module's header comment (right after the
    license line). When you change a module's role, update that line ‚Äî it's the
    smallest possible "doc in the same change" (see below). New first-party
    modules get a header synopsis automatically picked up on the next `make map`.
- **`doc/adr/`** ‚Äî append-only Architecture Decision Records for the *why* that
  the code and the map can't show. When you make a non-obvious design decision,
  add a new `NNNN-title.md` (Nygard template). **Never edit an accepted ADR** ‚Äî
  supersede it with a new one. See `doc/adr/README.md`.

## Keep docs in sync with code (in the SAME change)

When you add or change a **feature, fix, CLI option, or keyboard shortcut**,
update the relevant documentation as part of that same change ‚Äî never leave the
docs trailing behind the code. A behavior change with no doc update is
incomplete.

Doc surfaces ‚Äî check each that applies:

- **`README.md`** ‚Äî top-level feature/usage summary and the version line.
- **`guide/README.md`** ‚Äî the user guide (rendered through `guide/index.md`).
  Update the *Global Shortcuts* / *Playback* tables and add or extend a section
  when you introduce a feature.
- **`doc/ccutter.1`, `doc/ccutter.{fr,de,sv,fi}.1` and `doc/KEYBOARD.md` are
  GENERATED ‚Äî do not hand-edit.** For a CLI flag, add it to `cliOptions()` in
  `src/main.d` (the single source for `--help` and every man page); also add its
  translated help to each language table in `src/manpage.d` (English help is the
  fallback for any missing translation). For a shortcut, add a `com.shortcuts`
  `register(...)` entry (drives F12 help + the keyboard reference). Then run
  `make docs` (or `make -f Makefile.mac docs`) to regenerate all of them. The
  non-English man pages are machine-translated (a `.\"` note says so) ‚Äî flag a
  native-speaker review when strings change.
- **`doc/ARCHITECTURE.md` is GENERATED ‚Äî do not hand-edit.** Regenerated by
  `make map` (folded into `make docs`). To change what it shows, edit a module's
  one-line header synopsis, not the map. `make check-map` fails if it is stale.
- **`doc/ct2util.1`** ‚Äî still hand-maintained; update for ct2util CLI changes.
- **Version** lives only in the repo-root `Version` file. `com.util.APP_VERSION`
  string-imports it at compile time (build flags include `-J.`), so bumping the
  version = editing `Version` then rebuilding; it flows to `--help`, the man
  pages and the exported PRGs automatically. Because `APP_VERSION` is an enum
  (inlined), the Makefiles rebuild every user via `VERSION_USERS` ‚Äî add new
  `APP_VERSION` users to that list. (Top-level `README.md` still states the
  version in prose; update it by hand on a bump.)

There is no fully-automatic generator for the prose docs ‚Äî treat the list above
as the checklist to run through before considering a feature/fix done.

## Build note

The Makefiles have no inter-module dependency tracking: run `make clean` after
changing a base class or a string-imported asset (`src/c64/*.acme`,
`src/c64/player.bin`) so dependents are rebuilt.
